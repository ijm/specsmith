## Finite State Machine Control Pattern

### Summary

This specification defines a modular idiom for structuring execution flow using a classical finite state machine (FSM). The FSM is controlled by a stream of input symbols generated by a dedicated evaluator and handles transitions deterministically based on state and input. The intent is to provide a clean, reusable, and verifiable foundation for mode-driven or reactive applications.

### Scope and Intent

This module implements the control structure for a reactive or mode-based system using a formal FSM. The FSM governs application behavior by consuming input symbols derived from system state and external observations. The FSM transitions must be deterministic and well-scoped, enabling static reasoning and formal verification. The model supports exception-driven transitions and encapsulated state-local logic. This module assumes the application has separable concerns for evaluation (symbol generation) and action (effect application).

### Terminology and Conventions

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in BCP 14 (RFC 2119 and RFC 8174) when, and only when, they appear in all capitals. Ⓟ

**Domain-specific terms**:

- **FSM (Finite State Machine)**: A classical model consisting of states and deterministic transitions driven by input symbols. Ⓟ
- **Evaluator**: A module that observes internal and external environment state and emits symbols representing events or conditions. Ⓟ
- **Symbol**: An abstract, typed event or condition that triggers state transitions. Ⓟ
- **Gamma (`γ`) function**: A mapping from state to a transition function over input symbols. Ⓟ

### Functional Requirements

- The FSM module MUST define a function `gamma(state)` that returns a transition function for the current state. Ⓟ
- The transition function returned by `gamma` MUST be a partial function: `transition(symbol) -> (next_state, output)` for only valid input symbols in that state. Ⓣ
- The `Evaluator` module MUST emit symbols by observing relevant system variables and inputs. Ⓣ
- The FSM controller MUST call `symbol = evaluate(S)` to obtain the current symbol based on system state. Ⓟ
- The FSM controller MUST then call `transition = gamma(S)` and `S, output = transition(symbol)` to advance state. Ⓟ
- Transitions MUST be deterministic — for any valid `(state, symbol)` pair, the result MUST be a unique `(next_state, output)` pair. Ⓣ
- Invalid or undefined symbols for a state MUST raise an error or signal cleanly. Ⓣ
- Exceptions occurring elsewhere in the system MAY inject symbols into the FSM for handling recovery transitions. Ⓗ
- FSM transitions MAY emit output symbols to trigger downstream effects or actions. Ⓣ
- The FSM logic MUST be encapsulated in a separate module with a well-defined public interface. Ⓟ

### Structural Constraints

- Each FSM state MUST be encapsulated as its own transition function (e.g., via function, closure, or method). Ⓟ
- The transition map `gamma` MUST be structured as `state -> transition_function`. Ⓟ
- The FSM module MUST NOT contain business logic or system actions; these MUST be invoked only through emitted outputs. Ⓟ
- Evaluator logic MUST be isolated from transition logic and MUST NOT directly modify FSM state. Ⓟ
- Input symbols MUST be passed explicitly to the FSM and MUST NOT be globally observed. Ⓟ
- The FSM module MUST reside in its own file or namespace. Ⓟ

### Prohibited Elements

- FSM transitions MUST NOT be expressed as a global if-else or switch-case on `(state, symbol)` tuples. Ⓟ
- The evaluator MUST NOT encode control logic or state transitions. Ⓟ
- Input symbols MUST NOT be implicitly derived inside the FSM logic. Ⓟ
- Global state access inside transition functions is NOT ALLOWED. Ⓟ
- Transition logic MUST NOT be stateful outside of the `state` and `symbol` inputs. Ⓟ

### Verification

* **Ⓟ Provable**:

  * Check that FSM logic is encapsulated and that `gamma` maps states to functions.
  * Check that `evaluate()` is a separate function and that state is not directly mutated.
  * Confirm that FSM transitions are partial functions for input symbols.
  * Ensure no control logic appears outside `gamma` or its transition functions.

* **Ⓣ Testable**:

  * Provide test cases for each `(state, symbol)` pair and assert transition behavior.
  * Ensure undefined transitions raise or signal correctly.

* **Ⓔ Evaluatable**:

  * N/A for this module.

* **Ⓗ Heuristically Acceptable**:

  * Manual review can confirm separation of concerns, naming clarity, and clean factorization.
